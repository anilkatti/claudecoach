The Phase-2 coach: turn the profile-builder's signals into prioritized, evidence-cited, opt-in-apply recommendations.
Date: 2026-06-15 Status: Draft v2 — for your review Input: profile-builder v2 (production sensor) Build posture: production from scratch; prior art = ideas, not a dependency

1 · Problem & goal

profile-builder is a deliberate sensor: it collects evidence-verified signals and explicitly does not judge or recommend. Somebody has to turn those signals into action — /recommend-actions is that somebody: the Phase-2 "coach" that reads the profile and produces clear, prioritized, opt-in-apply recommendations for getting more out of Claude.

Your four original asks are the spine. This revision broadens each into a family of concrete actions (see §4), but the headline is still four moves:

bring inAcquire the right capabilities
Public skills, MCP servers, or plugins that fill a real gap the profile shows — and never one it can't point to.
tune setupTune your config surface
Trim bloat & dead weight · fill the always-on context with facts you keep re-typing · automate repetitive steps.
build assetBuild a reusable asset from a pattern
A recurring procedure becomes the lightest viable artifact — a memory line, a command, or a full skill.
work smarterAdopt better habits, drop worse ones
Match behavioral signals to best practices published by Anthropic & OpenAI — what to start, and what to stop.
Success criteria
Given a project's profile, emit actions.json + an HTML report where every action cites the signal it rests on, research-based actions cite a real dated source, impact is quantified where the data allows, and each action has an opt-in Apply step. No invented skills; no causal claims from correlational data.

7 · The agents — the multi-agent system

Topology: specialist fan-out → Opus synthesizer. Four specialists run in parallel (no shared state, no ordering), each owning one family and emitting candidate actions to a shared schema. A fifth reconciles across families and produces the final ranked set. All five default to Opus 4.8 — every call here is a judgment call (see the model-tier note below); profile-builder's Haiku-for-volume logic doesn't transfer, because this skill makes ~5 calls over a distilled profile, not one-per-session over transcripts. Each agent stays small & testable; web tools go only to the two research lanes.

① capability-scout
Bring in skills, MCP servers, or plugins that fill a real gap.
Model Opus 4.8 — gap-fit + existence-verify is trust-critical
Reads gaps, task_archetypes, domains, tools_and_materials, mcp_footprint − owned_capabilities
Tools curated capabilities_index.json + WebSearch/WebFetch (top-up & verify)
Emits acquire candidates: name, kind (skill/mcp/plugin), source, url, gap filled, freshness
Rail Never propose what it can't point to. Empty gap → "no public capability found; consider building one."
② config-doctor
Tune the local config: trim · fill · automate.
Model Opus 4.8 — drafting context/descriptions is judgment, not mechanics
Reads context_health + the config-relevant friction_signals
Emits trim (unused/dupes/dead MCP, token saving) · fill (capture reexplained facts → CLAUDE.md/memory) · automate (hook, permission allowlist, sharpen overlapping descriptions)
Rail "unused" = "unused in the sample." All edits reversible & consent-gated; quantify token impact from the real counts.
③ pattern-smith
Turn a recurring pattern into the lightest viable asset.
Model Opus 4.8 — creative synthesis of a new asset
Reads friction_signals, high-weight task_archetypes lacking an owner, rework
Emits a proposal + chosen form (memory line ▸ slash command ▸ full skill) with the pattern it resolves + evidence
Rail Only where the pattern recurs across ≥k sampled sessions; cite the count. Prefer the lightest form. Hands to skill-creator on apply — never writes the asset itself.
④ practice-coach
Start good habits, stop bad ones — cited.
Model Opus 4.8 — must not give generic or uncited advice
Reads behavioral_signals, friction_signals, habits (both polarities)
Tools curated best_practices.json + WebFetch (verify/refresh)
Emits adopt + stop recommendations: the change, expected effect, source + date
Rail Correlational only ("often alongside…", never "caused"). Every claim cites an Anthropic/OpenAI source with a freshness date.
⑤ action-synthesizer
Reconcile across families; produce the final ranked action set.
Model Opus — the cross-cutting call (same tier profile-builder uses for synthesis)
Reads all four candidate lists + the profile (for re-grounding) + index freshness
Does (a) dedupe; (b) resolve the artifact form — the headline reconciliation: one pattern might arrive as an install (scout), a hook (doctor), a skill (smith), or a context line — pick the lightest that works; (c) prioritize into do_now / consider / fyi from impact × confidence, with effort shown and impact quantified where the data allows; (d) enforce the evidence contract — drop any candidate that can't re-ground in the profile.
Emits the final actions.json (§10), incl. a not_recommended[] ledger so dropped candidates are visible.
Why a separate synthesizer
Specialists are intentionally blind to each other so each stays small and testable. Conflicts only exist across families (the same need can become an install, a hook, a skill, or a memory line), so resolving them — and choosing the lightest viable form — needs one agent that sees everything. It's also the single chokepoint for the honesty contract.
Model tiers — why all Opus ("is Haiku enough?")
All five agents default to Opus 4.8, exposed as one MODEL_TIER constant. profile-builder's tier split doesn't transfer: it runs Haiku per session across 15–30 sessions (high volume → economize), then Opus once for synthesis. recommend-actions is the opposite shape — ~5 calls total, each over the already-distilled profile (a few KB), and every one is a judgment call. At that volume the Haiku→Opus delta is ~$1–2/run (claude-api pricing, cached 2026-06-04: Opus $5/$25 vs Haiku $1/$5 per 1M tok) — negligible for an on-demand tool whose whole value is judgment quality. Sonnet ($3/$15) is a defensible middle for the two lighter lanes (scout ranking, config-doctor trims), but the specialists run in parallel and pattern-smith + the synthesizer are Opus regardless, so mixing tiers barely moves wall-clock or cost. Haiku's real home is a future high-volume "thorough mode" (Topology C), where per-candidate verification fans out into many small checks — with Opus still making the final call.

9 · Research mechanism (curated index + live top-up)

Two repo-shipped curated catalogs are the spine; live web is the top-up. This honors decision #3 and the rationale recorded in the prior coach design (which cites ToolRet arXiv:2503.01763 and arXiv:2406.17465 for "LLM-ranks-a-curated-index beats embedding retrieval on small tool/skill corpora" — as cited there, not independently re-verified here) while still genuinely "doing research."

Build time (offline, networked)

scripts/build_indexes.py — written fresh (the coach's build_skills_index.py is a reference, not a base). Fetches the public skills/plugin marketplace + Anthropic/OpenAI best-practice docs, normalizes, stamps built_at. Degrades gracefully: skip an unreachable source and log what was dropped — never ship a half-built index silently.

  • reference/capabilities_index.json — {name, kind: skill|mcp|plugin, source, one_liner, when_to_use, tags, url}
  • reference/best_practices.json — {id, principle, applies_to_signal, source_url, source_org, built_at}

Run time (local + optional live)

Ranking/matching runs against the curated catalogs (offline, always works). With network consent, the research lanes may:

  • Top-up — WebSearch for capabilities newer than built_at;
  • Verify — WebFetch to confirm a recommended capability/source still exists before it ships.

Every research-based action carries source + freshness.

Network consent
The run opens with a consent gate. Curated path is fully local; live top-up is the only network use and is announced up front. Decline → curated-only run, flagged as such in the report.

11 · Output & the Apply engine

actions.html — the report

Deterministic render.py (same spirit as visualize.py), styled to match profile.html. Sections: Headline (top "do now", profile freshness, network flag) · Do now / Consider / FYI action cards (rationale, evidence quote, quantified impact, source+freshness, Apply affordance) · Context-health savings (before/after token estimate) · Considered but not recommended (the ledger) · Fine print (nondeterminism, sampling caveat, index dates).

Apply engine — opt-in, per action

apply.kindWhat "Apply" doesUsed bySafety
run_commandShow the exact install command, run on yes.install_skill, add_mcp, add_pluginCommand shown before it runs.
edit_fileCapture context / trim CLAUDE.md / memory — present a diff, back up, apply on yes.capture_context, trim, author (memory form)Diff + backup; reversible.
scaffold_skillHand the drafted spec to skill-creator.author_asset (skill/command form)skill-creator owns authoring + review.
handoff_skillConfig change via the owning skill.automate_hook → update-config; cut_permission_friction → fewer-permission-prompts; merge_sharpen → skill-creator (description optimize)Owning skill enforces settings safety.

Reversibility
"Remove a capability" = archive/unlink, never destroy. The report names exactly what's affected; anything used in a sampled session is flagged before removal is offered.

12 · Honesty rails & exec principles

Evidence or it doesn't ship
Every action cites the profile signal (and verified quote) it rests on. The synthesizer drops any candidate that can't re-ground.
Never invent a capability
Recommend only skills/MCP/plugins in the index or confirmed live. A gap with no match → "consider building one," never a hallucinated name.
Quantify impact (exec lens)
Lead with the number when the data allows — "saves ~1.3k always-on tokens/prompt," "avoids re-explaining X (seen in 6/15 sessions)." Use the real context_health counts; stay qualitative only when you must.
Lightest viable artifact (exec lens)
Prefer a CLAUDE.md line over a command over a skill. Don't build a skill when a one-liner does the job — matches your own "simplicity first" rule.
Correlational, not causal
Habit/best-practice findings say "often alongside…," never "caused." Each carries a counted evidence string.
Freshness & sampling shown
Research actions show source + dates. "Unused" means "unused in the sample" — never "globally dead."
Consent twice
Network use consented up front; each apply consented individually. Read-only, non-networked by default.
Reversible & stale-aware
Archive/unlink, never delete; back up + diff before editing context files. If the profile is stale, offer to refresh first.

15 · Testing strategy

  • Deterministic plumbing → pytest: load_profile.py (slug, missing/stale profile, lane split), render.py (stable HTML/console from a fixed actions.json), apply.py (backup, diff, archive/unlink round-trip — and refusal to delete), build_indexes.py (graceful source-drop, built_at).
  • Fixtures: a profile-builder-v2 fixture for a developer and a non-developer (the accountant fixture profile-builder already uses) so audience-neutrality is tested.
  • Contract tests: every action has non-empty evidence; no acquire action lacks a source.url/index ref; every removal is reversible:true; every research action has a freshness.
  • LLM steps are nondeterministic — not asserted on exact text; snapshot the shape of a synthesizer output against a recorded fixture.
